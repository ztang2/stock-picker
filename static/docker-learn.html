🤔为什么需要 Docker？

一句话："在我电脑上能跑"不代表"在别人电脑上也能跑"。

❌ 没有 Docker

  • 新人入职，花1天配环境
  • "我这里能跑啊" — 经典甩锅
  • Python 3.9 vs 3.11 兼容问题
  • macOS vs Linux 行为不同
  • 装依赖到一半报错
  • 手动改配置忘记记录

✅ 有 Docker

  • 新人入职，一条命令搞定
  • 你跑的和我跑的100%一样
  • Python版本锁定在容器里
  • 不管什么系统，容器内都是Linux
  • 依赖已经装好在镜像里
  • Dockerfile就是配置文档

📦Docker 三大概念

Docker 只有三个核心概念，理解了就会了：

📄 Dockerfile
菜谱
→ build →
📦 Image
预制菜
→ run →
🏃 Container
端上桌的菜

1. Dockerfile（菜谱 📄）

一个文本文件，写着"怎么打包"的步骤。

就像菜谱告诉你：先放油，再放蒜，炒3分钟...

你只需要写一次。改代码不用改它。

2. Image（镜像/预制菜 📦）

docker build 按照 Dockerfile 的步骤打包出来的东西。

它是一个静态快照——包含了操作系统 + Python + 所有库 + 你的代码。

就像预制菜：做好了、封装了、冻在冰箱里。本身不会运行。

可以分发给任何人。他们不需要知道你怎么做的，拿来直接加热（run）就行。

3. Container（容器/端上桌的菜 🏃）

docker run 把 Image 变成一个正在运行的进程。

就像把预制菜从冰箱拿出来，加热，端上桌。

一个 Image 可以同时启动多个 Container（就像一个菜谱可以做很多盘菜）。

Docker vs VM（虚拟机）

🐳 Docker Container

  • 共享宿主机内核（轻量）
  • 启动：秒级
  • 体积：MB级（几百MB）
  • 资源占用：极小
  • 适合：部署应用

🖥️ 虚拟机 (VM)

  • 完整操作系统（重量级）
  • 启动：分钟级
  • 体积：GB级（几个GB）
  • 资源占用：很大
  • 适合：跑不同OS

🖥️ VM

📄我们的 Dockerfile — 逐行解释

这是 stock-picker 的 Dockerfile，一共8行。点击每一行看详解：

FROM python:3.11-slim 👈 点击看详解

选基础镜像 — 就像选一台"新电脑"的配置。

python:3.11-slim = 装了 Python 3.11 的精简版 Linux。

为什么 slim？完整版800MB，slim只要150MB。我们不需要C编译器等额外工具。

💡 其他选项：python:3.11(完整版), python:3.11-alpine(超精简但兼容性差)

WORKDIR /app 设置工作目录

在容器里创建并进入 /app 目录。

后面所有 COPY、RUN 命令都在这个目录下执行。

就像你打开 Terminal 先 cd ~/clawd/stock-picker 一样。

COPY requirements.txt . 先复制依赖清单

只复制 requirements.txt 到容器里。

为什么不直接 COPY 所有文件？因为 Docker 有缓存机制：

如果 requirements.txt 没变 → Docker 跳过下一步的 pip install → build 从几分钟变成几秒

如果你改了代码但没加新库，pip install 这步直接用缓存，不重新装。

💡 这是 Dockerfile 的最佳实践：把"不常变"的步骤放前面。

RUN pip install --no-cache-dir -r requirements.txt 安装 Python 依赖

在容器里执行 pip install。

--no-cache-dir = 不保存 pip 的下载缓存，减小镜像体积。

-r requirements.txt = 按清单装所有库（FastAPI, pandas, XGBoost...）

这一步最耗时（约60秒），但因为缓存机制，改代码后不会重新跑。

COPY src/ src/ 复制源代码

把你的 Python 代码复制到容器的 /app/src/ 目录。

每次改代码 → 重新 build → 这一步会更新 → 但前面的 pip install 用缓存跳过。

所以改代码后重新 build 只要几秒，不用重新装依赖。

COPY static/ static/  &&  COPY config.yaml .  &&  COPY data/ data/ 复制其他文件

把网页文件（static/）、配置（config.yaml）、数据（data/）也复制进去。

容器是隔离的——它看不到你 Mac 上的文件。必须明确 COPY 进去。

EXPOSE 8000 声明端口

告诉 Docker "这个容器会用 8000 端口"。

注意：这只是一个声明/文档，不会自动打开端口。

真正打开端口要在 docker run -p 8002:8000 里指定。

CMD ["python", "-m", "uvicorn", "src.api:app", "--host", "0.0.0.0", "--port", "8000"] 启动命令

容器启动时执行这条命令。

等同于你在 Terminal 里跑 python -m uvicorn src.api:app --host 0.0.0.0 --port 8000

--host 0.0.0.0 = 监听所有网络接口（不只是 localhost），这样容器外面才能访问。

💡 CMD vs RUN 的区别：RUN 在 build 时执行，CMD 在 run 时执行。

不加 -d 会怎样？

docker run（前台）

  • 日志直接打印在Terminal
  • Ctrl+C 停止容器
  • 关掉Terminal = 容器停了

  • 关掉Terminal容器还在跑
  • 适合：正式运行

🔗端口映射 — 最容易搞混的概念

容器是隔离的。它有自己的网络。你在浏览器输 localhost:8000，请求到不了容器里面。

需要用 -p 建一个"隧道"：

-p 8002:8000

↑ ↑
你Mac的 容器内的
端口 端口
💻
你的 Mac
localhost:8002
🔗 → → →
🐳
Docker 容器
0.0.0.0:8000

为什么我们用 8002 不用 8000？

因为你 Mac 上已经有一个 stock-picker 在 :8000 跑着（直接跑的 Python 进程，不是 Docker）。

同一个端口不能两个程序用 → 所以 Docker 用 8002 避免冲突。

常见端口映射

-p 8000:8000 Mac 8000 → 容器 8000（最直觉）
-p 8002:8000 Mac 8002 → 容器 8000（避免冲突）
-p 80:8000 Mac 80 → 容器 8000（这样输 localhost 不用加端口号）
-p 3000:3000 -p 8000:8000 可以映射多个端口

不加 -p 会怎样？

容器跑了但你访问不了。请求到不了容器里。这是新手最常踩的坑。

面试常问

Q: CMD 和 ENTRYPOINT 的区别？

A: CMD 可以被 docker run 后面的参数覆盖，ENTRYPOINT 不会。一般用 CMD 就够了。


Q: COPY 和 ADD 的区别？

A: ADD 能自动解压 tar 文件和下载 URL。但官方推荐用 COPY（更明确）。


Q: 怎么减小 Docker 镜像体积？

A: 用 slim/alpine 基础镜像，合并 RUN 命令，用 multi-stage build，.dockerignore 排除无用文件。


Q: Docker 网络模式有哪些？

A: bridge（默认，容器间隔离）, host（共享宿主网络）, none（无网络）。

📚进阶概念 — 碰到时再看

你目前的知识够用了。以下三个概念等实际碰到再深入。

1. Docker Compose 🎼

同时管理多个容器。现实中一个项目不只有API，还有数据库、缓存等。

比如 stock-picker 如果加了数据库：

# docker-compose.yml
version: '3'
services:
  api:
    build: .
    ports:
      - "8000:8000"
    depends_on:
      - db

  db:
    image: postgres:15
    environment:
      POSTGRES_PASSWORD: secret
    volumes:
      - db-data:/var/lib/postgresql/data

volumes:
  db-data:
docker compose up -d 一条命令启动所有容器（API + 数据库）
docker compose down 一条命令停止并清理所有容器
docker compose logs -f 查看所有容器的日志（实时跟踪）

面试常问："你的项目有多个服务怎么管理？" → "Docker Compose，一个 YAML 文件定义所有服务和依赖关系。"

2. Volume（数据卷）💾

问题：容器删了，里面的数据也没了。

比如数据库跑在容器里，你 docker rm 后所有数据丢失。

Volume 把数据存在容器外面（你的 Mac 硬盘上），容器删了数据还在。

🐳
容器（临时的）
删了就没了
🔗
💾
Volume（持久的）
存在 Mac 硬盘上
# 创建一个 volume 并挂载到容器
docker run -v my-data:/app/data stock-picker

# ↑ volume名 ↑ 容器内路径
# 容器里 /app/data 的内容会持久保存在 my-data 这个 volume 里
docker volume ls 列出所有 volume
docker volume rm 名字 删除 volume
-v ~/data:/app/data 也可以直接映射 Mac 上的目录（bind mount）

3. Docker Hub 🌐

像 GitHub，但存的是 Docker 镜像而不是代码。

💻 你的电脑
docker build
→ push →
🌐 Docker Hub
hub.docker.com
→ pull →
☁️ 服务器/同事
docker run
# 给镜像打标签（用你的 Docker Hub 用户名）
docker tag stock-picker ztang2/stock-picker:v1

# 推送到 Docker Hub
docker push ztang2/stock-picker:v1

# 别人（或服务器）拉取并运行
docker pull ztang2/stock-picker:v1
docker run -p 8000:8000 ztang2/stock-picker:v1

你之前用的 python:3.11-slim 就是从 Docker Hub 下载的官方 Python 镜像。

什么时候学这些？

Docker Compose 当你的项目需要数据库或多个服务时
Volume 当你需要在容器里存持久数据时
Docker Hub 当你想把镜像分享给别人或部署到服务器时

1. Docker 的三个核心概念是？

Container, VM, Cloud

3. -p 8002:8000 左边的 8002 是？

容器内的端口
你 Mac（宿主机）的端口
Docker 的默认端口
格式是 宿主机端口:容器端口。左边是你 Mac 上的端口，右边是容器内部的端口。

4. 为什么先 COPY requirements.txt 再 COPY src/？

文件大小排序，小的先复制
没有原因，顺序无所谓
利用缓存：依赖不变时跳过 pip install
Docker 每一层（每条指令）都会缓存。如果 requirements.txt 没变，pip install 那步直接用缓存。这样改代码后 rebuild 只要几秒。

5. Container 和 VM 最大的区别是？

Container 共享宿主内核，VM 有独立内核
Container 更安全
VM 更快
Container 直接用宿主机的 Linux 内核，所以体积小（MB级）、启动快（秒级）。VM 包含完整操作系统，体积GB级，启动分钟级。

6. 容器停了之后，localhost:8002 还能访问吗？

能，数据还在
不能，容器停了端口就关了
看情况
容器停了 = 进程结束了 = 端口关了。需要重新 docker run 才能访问。但 Image 还在，随时可以重新启动。
